# Case Study · 我打造了自己的工具收藏宇宙

> "那个网站叫啥来着？" — 一个普通日常的痛点,我用 AI 把它做成了仪式感的入口。

---

## 痛点

我每天会看 5–10 个新工具站、灵感站、AI 产品。两年下来浏览器收藏夹有 1000+ 条。

**但我从来找不到我要的那个**。

```
"我之前收过一个能批量去除视频背景的 AI 工具,叫啥来着?"
"那个有手绘动效的复古设计灵感站,URL 我忘了"
"做出海产品的 API 中转站,哪家最稳来着?"
```

收藏夹只是把书签堆在那里 —— 它**不理解你为什么要收藏这个**,也**没法帮你找回**。

---

## 解法不是"再做一个收藏夹"

我用了一个月,做了一套属于我自己的工具宇宙。**核心理念**:

> 我已经有一个最熟练的收藏动作 — `⌘D`。
> 我需要的不是新工具,而是一个 **"⌘D 之后" 的智能管家**。

按下 `⌘D`,后台帮我做这些事:
1. 截图
2. 用 AI 读懂网站 — 它能干什么、适合谁、有没有同类
3. 自动分类(我手懒,从来不分类)
4. 写到我的 Notion (主存储) + Obsidian (本地备份) + 工具墙(展示)
5. 帮我建立"找不到时的检索维度":同义词、别名、替代品、搭配品

我做的不是收藏夹,是**记忆助手**。

---

## 仪式感比功能感重要

工具墙做出来后,我意识到一个问题: **如果它只是"干净的卡片网格",那我打开它就只是打开一个数据库,没什么"我的"东西**。

我想要的是一个**有人格的入口** —— 让我每次打开,都觉得"这是我的 corner"。

所以入口我做了一张可以**真的撕开**的纸。

```
打开网站 → 一张深色纸盖住一切,大字写着 "那个网站叫啥来着?"
鼠标拖动 → 真的撕扯,出现真实的洞,不是变形
撕开第一层 → 透过洞看见第二层 (米色 + 我的品牌)
再撕一次 → 工具墙出现
```

**用户主动撕开 = 一种小小的 ritual = 让人记住你的网站**。

技术上这是 Three.js + Verlet 物理 + 多层 cloth + 真洞渲染(撕掉的三角形从 mesh 里跳出来,不是变形)。

我打磨这个撕扯感打了 25 轮,最后把参数和算法洞察沉淀成一个 [独立 Skill 上 GitHub](https://github.com/Jane-xiaoer/claude-skill-tearable-cloth)。

---

## 三个我自己最骄傲的设计决策

### 1. 浏览器层只触发,不绑定

很多"超级收藏夹"工具会要求你装它的浏览器扩展。我**坚决不这样**。

我的解法: 后台守护监听任何 Chromium 系浏览器的书签文件 (FSEvents 内核级监听)。任何浏览器 — Chrome、Brave、Edge、Tabbit、Doubao、Comet、Quark、Dia — 第一次按 `⌘D`,系统**自动发现并加入监控**,无需配置。

这意味着我以后装任何新浏览器,都不用动代码 — 只要它是 Chromium 系。

### 2. AI 不只是分类,而是建立"检索维度"

普通分类只解决了"放进哪个文件夹"。它没解决"我想找的时候搜不到"。

我让 AI 给每个工具生成 5 个维度的检索标签:
- **能做什么** (capabilities) — 任务型查询命中
- **适合什么场景** (scenarios) — 场景型查询命中
- **同义词/别名/口语** (search_keywords) — 模糊回忆查询的命脉
- **替代谁 / 类似谁 / 配谁用** (alternatives) — 比较型查询
- **小耳备注** (我的真实判断) — 推荐时引用

这样我搜"能批量去视频背景的"、"做品牌时用什么"、"v0 替代什么",AI 都能从 348 件里精准捞出来。

### 3. 公开但防白嫖 — 三层鉴权

工具墙公开,任何人都能看,**任何人都能用 AI 推荐**。

但 AI 推荐每次调 Gemini 是要花钱的。我做了三层鉴权:

```
Free 用户 (默认)        — 我请客,3 次/天 (IP+cookie 指纹防绕过)
BYOK 用户                — 你填自己的 Gemini key,无限,你出钱
Master 用户 (我自己)     — 填密码,无限,我出钱
```

公开站这事的难点不是技术,是**心态** — 既不能完全闭门私用,也不能让陌生人薅我钱包。这套架构两边都满足了。

## 沉淀

我没有让这事就停在"我自己用得爽"。

我把整个项目打包成 [`xiaoers-arsenal`](https://github.com/Jane-xiaoer/xiaoers-arsenal):
- **README** 讲故事 (你正在读的这个文件)
- **ARCHITECTURE** 讲数据流 / 技术选型决策
- **OPERATIONS** 讲运维手册
- **CASE_STUDY** 讲品牌故事
- **CLAUDE.md** 讲怎么让未来的 AI 进来快速理解上下文

把撕扯入口单独抽出来,做成 [独立 skill 上 GitHub](https://github.com/Jane-xiaoer/claude-skill-tearable-cloth) — 别人想给自己网站加这种入口,直接装这个 skill,Claude Code 一句话帮做好。

## 这事教会我的

1. **痛点要够真实** — 我不是想做产品,我是真的找不到我两年前收藏的东西。这个项目的所有决策都是从这个痛点反推的。

2. **AI 不是新功能,是新基建** — 加 AI 不是"在原来的产品上加一个 chat 框",而是把 AI 嵌到流程里 (⌘D → 自动分析)。前者是噱头,后者是 leverage。

3. **仪式感 > 功能感** — 工具墙没有撕扯入口的时候它只是"网站",有了之后它变成"我的 corner"。10 秒的体验差异,记忆点完全不同。

4. **打包是产品的一部分** — 一个项目不是"代码跑起来就行",还要有可分享的 README、可继承的架构文档、可沉淀的 skill。一个项目做完不打包 = 浪费了 80% 的可重用价值。

5. **Claude Code 是个加速器,不是替代品** — 我跟 Claude 协作了 ~25 轮才把撕扯感调对。它能 80% 帮我对齐方向,剩下 20% 还是要我自己去交互、感受、判断。
